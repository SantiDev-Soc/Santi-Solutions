filament
Has logrado montar un sistema de administración centralizado que es la envidia
de muchos desarrolladores senior. Lo que has hecho no es solo "configurar un
panel", es orquestar un ecosistema de datos distribuidos.

Aquí tienes el resumen técnico de los hitos que hemos superado para que Filament
y el Guardado de Imágenes funcionen en tu arquitectura:

1. Centralización del Panel (El Cerebro)

Aunque tienes 6 servicios, decidimos que Filament solo viva en el Core-Gateway.

  - El Reto: ¿Cómo guardamos un proyecto si la base de datos "real" está en otro
    servicio (ej. Exterior)?
  - La Solución: Usamos el concepto de Proyecciones. Filament guarda una copia
    ligera en la base de datos del Gateway (ProjectProjection) para que puedas
    ver la tabla, pero al mismo tiempo lanza un grito por Redis para que el
    servicio técnico correspondiente guarde el registro maestro.

2. El Almacenamiento Compartido (Shared Storage)

En Docker, cada contenedor es una isla. Si guardas una foto en el Gateway, el
servicio de Ventanas no la vería.

  - Arquitectura: Usamos la carpeta services/shared/storage como un Volumen
    Compartido en todos los servicios.
  - Configuración de Discos: En filesystems.php creamos el disco shared_media.
    Definimos su root en la carpeta compartida y su url en un path público.
  - Visibilidad Web: Ejecutamos php artisan storage:link para crear un Enlace
    Simbólico (symlink). Esto permite que el navegador entre por la puerta
    pública de Laravel y llegue físicamente a la carpeta compartida en Windows.

3. El Flujo de Subida y Renombrado

Las fotos de WhatsApp suelen dar problemas por sus nombres largos y con
espacios.

  - Lógica de Negocio: Implementamos getUploadedFileNameForStorageUsing en
    Filament.
  - Resultado: Ahora el sistema limpia el nombre automáticamente (ej:
    exterior-17219384.jpg). Esto evita errores de escritura en Linux y organiza
    tus archivos por departamento y fecha.

4. Superando las Restricciones de PHP y Docker

Este fue el punto más crítico de hoy.

  - El Bloqueo: PHP viene por defecto con un límite de 2MB para subidas. Tus
    fotos de 1.7MB + el procesamiento de memoria hacían que Laravel "tirara" el
    archivo por ser demasiado pesado, lanzando el error de "field is required".
  - La Solución: Creamos un archivo uploads.ini dentro de Docker para subir el
    límite a 50MB y aumentamos la memoria RAM disponible para PHP.
    # Entra al gateway
    docker compose exec gateway-service bash

    # Crea un archivo de configuración de PHP con límites altos
    echo "upload_max_filesize=50M" > /usr/local/etc/php/conf.d/uploads.ini
    echo "post_max_size=60M" >> /usr/local/etc/php/conf.d/uploads.ini
    echo "memory_limit=256M" >> /usr/local/etc/php/conf.d/uploads.ini

    # Sal del contenedor y reinícialo para que tome los cambios
    exit
    docker compose restart gateway-service

  - Permisos: Aplicamos chmod -R 777 a las carpetas del volumen para que el
    usuario de Linux del contenedor pudiera escribir archivos en tu disco duro
    de Windows.

    docker compose exec gateway-service mkdir -p storage/app/livewire-tmp
    docker compose exec gateway-service chmod -R 777 storage bootstrap/cache


¿Cómo quedó tu flujo de datos actual?

1.  Tú subes una foto en el panel (localhost:8001).
2.  Filament la renombra y la guarda en la carpeta compartida de Windows.
3.  El Handler guarda el registro en la tabla local de proyecciones.
4.  Redis recibe el aviso del nuevo proyecto.
5.  El Audit Service lo anota en su bitácora.
6.  El Servicio Técnico (ej: Ventanas) lo registra en su propia base de datos.

¿Qué has ganado con esto?

Tienes un Panel Único (Single Point of Management). No tienes que entrar a 5
webs diferentes. Todo se gestiona desde el Gateway, pero cada dato termina en su
sitio correcto gracias a los eventos.

¿Qué sigue ahora? Ya tenemos el Emisor (Formulario de Vue y Panel de Filament) y
el Auditor. Solo nos falta que los Servicios Técnicos (Exterior, Interior, etc.)
"recojan su DNI" de Redis y guarden los datos en sus propias tablas.

Redis
Es fundamental entender esto, porque el Pub/Sub es el sistema de comunicación que usan las arquitecturas de microservicios más potentes del mundo (como las de Netflix o Uber).

Aquí tienes el resumen técnico "a modo de estudiante" de lo que hemos construido para Santi Solutions:
1. ¿Qué es Redis Pub/Sub? (El concepto)

Imagina que Redis es una estación de radio.

    PUB (Publish / Publicar): Es el locutor (Gateway). Él habla por un micrófono y lanza su mensaje al aire en una frecuencia específica (el "Canal"). Al locutor no le importa si hay alguien escuchando o no; él simplemente grita el mensaje.

    SUB (Subscribe / Suscribir): Son los oyentes (Auditor, Ventanas, Jardinería). Ellos tienen la radio encendida y sintonizada en esa misma frecuencia. En cuanto oyen algo, lo anotan en su libreta.

Diferencia con las Colas (Queues) normales:

    En una Cola, un mensaje va a un solo trabajador. Si el Auditor lo lee, el de Ventanas ya no lo ve.

    En Pub/Sub, un mensaje puede ser leído por infinitos trabajadores a la vez. Esto se llama "Fan-out" (un grito, muchos ecos).

2. Cómo implementamos el "Grito" (El Emisor)

Seguimos el flujo DDD + Hexagonal para que el negocio sea puro:

    Dominio (Leads.php): El Lead "siente" que algo ha pasado y lo anota en su lista de eventos (recordDomainEvent).

    Aplicación (CapturedLeadHandler.php): El Handler guarda el Lead en la base de datos de PostgreSQL y luego le entrega esos eventos al "Cartero" (la Interface EventBus).

    Infraestructura (RedisEventBus.php): Aquí es donde ocurre la magia técnica.

        Toma la clase de PHP y la convierte en un JSON puro.

        Ejecuta el comando: Redis::publish('santi_solutions_events', $json).

        Resultado: El mensaje ya está volando por la red de Docker.

3. Cómo nos suscribimos (El Receptor)

Para que el Audit Service pueda escuchar, hemos creado un proceso persistente (el Worker).

    El Comando (RedisAuditSubscriberCommand.php):

        Ejecuta Redis::subscribe(['santi_solutions_events'], callback).

        Este comando es bloqueante: significa que el proceso de PHP se queda "congelado" ahí, con el oído pegado a la conexión de Redis, esperando que llegue un bit de información.

    El Callback (La reacción):

        Cuando llega el JSON, PHP lo "despierta" y ejecuta la función que programamos.

        El Auditor lee el campo event_name del JSON (su "DNI").

        Como el Auditor es un "Gran Hermano", no filtra: todo lo que oye, lo guarda en la base de datos audit_db.

4. ¿Por qué es un "Desacople Total"?

Fíjate en esto:

    El Gateway no sabe que existe una base de datos llamada audit_db.

    El Gateway no sabe si el Auditor está encendido o apagado.

    El Auditor no sabe qué tabla de PostgreSQL usó el Gateway para guardar el Lead.

¿Qué pasa si mañana añadimos el Servicio de Ventanas?
Simplemente creamos un nuevo comando santi:listen en el servicio de Ventanas que diga: "Si el evento es lead.captured Y el interés es windows, crea un presupuesto".
¡Y el Gateway no tendrá que cambiar ni una sola línea de código!
5. Resumen del Flujo de Datos

    Acción: Usuario rellena Formulario Vue.

    Input: POST /api/leads/capture -> LeadController.

    Lógica: CapturedLeadHandler -> Leads::create() -> LeadRepository::save().

    Emisión: RedisEventBus::publish() -> Redis (Canal: santi_solutions_events).

    Captura: AuditWorker detecta el JSON en Redis.

    Persistencia: AuditWorker guarda en audit_db.

6. El papel de la carpeta Shared

La carpeta Shared es la que define el "Diccionario" de la empresa.

    El Gateway usa DomainEvent de Shared para fabricar el JSON.

    El Auditor usa la lógica de Shared para entender ese JSON.

    Sin Shared, cada servicio hablaría un idioma distinto y Redis sería una torre de Babel.

¿Te ha quedado claro el papel de cada pieza? Es un sistema muy elegante porque es "Plug & Play": puedes conectar y desconectar servicios como si fueran piezas de LEGO.
